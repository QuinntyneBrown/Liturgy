What Liturgy should borrow from Linear

Agents that keep the rhythm

Linear's deepest idea isn't a feature — it's a stance: agents are first-class members of the workspace, but they are delegates, never owners. Liturgy is unusually well-placed to adopt it, because the board already has a server-authoritative rule engine. The liturgy itself becomes the guardrail that teaches agents how to work.

Field notes

How Linear lets agents work

01 · IDENTITY
Agents are members

Each agent authenticates as itself (OAuth actor=app) with its own token and scoped permissions — app:assignable, app:mentionable. It appears in assignee dropdowns, clearly badged as an agent, never impersonating a human.

02 · ACCOUNTABILITY
Delegation, not assignment

When work goes to an agent, a human stays the assignee — accountable for the result — while the agent is added as a delegate who does the work. Linear's rule: “agents cannot be held accountable.”

03 · TRANSPARENCY
Sessions with visible state

Every agent interaction is a session — waiting, working, completed, errored — with structured activities: thoughts, tool calls, and elicitations (questions back to humans). Instant acknowledgment is a design principle.

04 · ACCESS
A standard tool surface

An MCP server exposes the product as tools — list, create, update, move, assign, comment — so any MCP-capable agent (Claude, Cursor, Codex…) can operate the board without bespoke integration.

Rather than direct assignment — which implies accountability — agents are delegated work. Issues retain human assignees, who remain accountable, while agents handle delegated tasks. Linear — Agent Interaction SDK design notes

Track A · The headline

The agent-operable board

Seven features, in dependency order, that let an agent pick up a card, move it through the columns, log its movements — and stay legible to the humans it serves.

A1 · FOUNDATION

Agent members & machine auth

Introduce agent identities as workspace members: created by a Lead, badged as agents everywhere they appear, authenticated with revocable API keys or personal access tokens instead of passwords. Scope them the way Linear scopes OAuth apps — assignable and mentionable are opt-in grants, not defaults.

Builds on: JWT issuer & Membership model — adds a credential type and an IsAgent member kind. Everything else in this track depends on it.
A2 · ACCOUNTABILITY

Delegation, never ownership

Keep the human as the card's assignee; add a delegate field for the agent doing the work. This is Linear's cleanest idea and it fits Liturgy's theology exactly: an agent may Render, but a human still Rejoices. The 5R loop keeps a person accountable for every card by design.

On the card: QB assignee ◇ claude · delegate
A3 · ACCESS

An MCP server for the board

Expose the existing board API as MCP tools — get_board, create_card, move_card, assign_card, point_card, log_movement, mark_done — so Claude Code or any MCP client can operate Liturgy directly. The enforcement engine already rejects illegal moves server-side; the 409 is the guardrail. The engine teaches agents the liturgy.

Builds on: BoardController + LoopController endpoints (they already exist 1:1) and A1 for authentication.
A4 · TRANSPARENCY

Agent sessions & the activity feed

Every delegation opens a session with a visible state — waiting · working · completed · errored — and writes structured activities: what the agent thought, which tools it called, what it asked. Pair this with a per-card activity log (currently absent) so humans can audit any change, agent or not.

New: AgentSession + ActivityEntry entities; the audit log benefits every feature in Track B too.
A5 · CONVERSATION

Comments & @mentions

Cards need a thread. Comments give agents a place to report progress and ask questions (Linear's “elicitations”), and @mentioning an agent becomes the second trigger besides delegation. Mentions of humans feed a notification inbox later.

New: Comment entity + SignalR CommentAdded event; mention parsing triggers A6 webhooks.
A6 · TRIGGERS

Webhooks out

SignalR pushes to browsers, but agents live elsewhere. Outbound webhooks — card.delegated, agent.mentioned, card.moved, gate.opened — with signed payloads let an agent wake up when work arrives, exactly how Linear's agent sessions are triggered.

Builds on: IRealtimeNotifier seam — add a webhook dispatcher beside the SignalR notifier.
A7 · PRESENCE

Agents visible on the board

When an agent holds a card, show it: a pulsing live badge on the work card, the agent's mark beside the assignee avatar, and a toast when it finishes. The design system already has the vocabulary (.badge--live, .toast) and SignalR already has per-project groups — this is UI wiring, not new infrastructure.

Track B · Fundamentals

Board features Linear gets right

Less glamorous, but several are prerequisites — an agent can't “use the board” well if cards can't be edited, ordered, or prioritized.

B1

Card editing

There is no endpoint to change a title or description after creation. Agents (and humans) need PATCH /cards/{id} before anything else.

B2

Manual ordering

Columns sort by creation date only, so drag order doesn't survive a reload. Add a persisted rank (LexoRank-style) and a reorder endpoint — the board's biggest structural gap.

B3

Priority & labels

Urgent → Low priority plus workspace labels. This is how humans tell agents what to pick up next — “take the top of the queue” needs a queue.

B4

Triage inbox

Agent- and intake-created cards land in a triage lane for a human to route — Liturgy's version of Linear's Triage, and a natural home for the Request movement.

B5

Real card relations

Blocked-by, relates-to, sub-cards. Today “blocked” is a bare flag with no cause — agents can't reason about what unblocks a card.

B6

Command palette

Linear's signature Cmd+K — every board action (move, assign, point, delegate) as a keyboard command. Bonus: the same action registry can back the MCP tools.

B7

Cycle analytics

Sprints already exist — add cycle-time and throughput per sprint, split human vs. agent, echoing Linear's per-agent cycle metrics and the dashboard's momentum stats.

Order of operations

A three-phase path

PHASE 1 · FOUNDATIONS
Make the board machine-ready

Agent members + API keys (A1) · card edit endpoint (B1) · persisted ordering (B2) · activity log (A4's audit half). Nothing agent-visible yet — everything agent-necessary.

PHASE 2 · THE AGENT SURFACE
Let agents in

MCP server (A3) · delegation field (A2) · agent sessions (A4) · comments + @mentions (A5) · webhooks (A6). At the end of this phase, Claude can pick up a card, work the loop, and move it — with a human still accountable.

PHASE 3 · THE EXPERIENCE
Make it feel alive

Agent presence on the board (A7) · priority & labels (B3) · triage lane (B4) · relations (B5) · command palette (B6) · human-vs-agent cycle analytics (B7).

The enforcement engine is the reason this works: an agent cannot skip the loop, cannot force Done, cannot break the rhythm — the server won't let it. Liturgy doesn't need to trust its agents. It needs only to let them pray in order. Thesis of this brief
