# UX — User Experience: L2 Requirements

Detailed requirements for the [User Experience capability](./L1.md). Each requirement traces to one L1 in this capability. Acceptance criteria are in Given/When/Then form; acceptance test files must carry a `Traces to: L2-UX-###` comment header.

---

## L2-UX-001: FaithTech color tokens
**Traces to:** L1-UX-001
**Source:** implementation + mocks

The UI uses the FaithTech palette as CSS custom properties — ink `#16160c`, paper `#ffffff`,
warm sand greys, and the signature lime `#c6fb50` — and does not invent colors.

**Acceptance Criteria:**
1. Given any screen, when inspected, then core surfaces use `--ink #16160c` and `--paper #ffffff` and sand greys `#f5f0f0`/`#e4e0d8`/`#c6c5bb`.
2. Given the signature lime `#c6fb50`, when used, then it appears only on the active phase, the current step, and the single primary action of a screen (never decorative).


## L2-UX-002: Phase color coding
**Traces to:** L1-UX-001
**Source:** implementation + mocks

Each 4D phase has a fixed color: Discover `#1d8fb9`, Discern `#f05228`, Develop `#32a432`,
Demonstrate `#ffb300`.

**Acceptance Criteria:**
1. Given a phase badge or lane, when it renders, then it uses that phase's assigned color.
2. Given the four phases across dashboard, journey, and board, when rendered, then phase coloring is consistent everywhere.


## L2-UX-003: Typography hierarchy
**Traces to:** L1-UX-001
**Source:** implementation + mocks

Three typefaces carry fixed roles: Hanken Grotesk (display/headings), Inter (body), Newsreader
italic (scripture/reflection only).

**Acceptance Criteria:**
1. Given headings and eyebrows, when rendered, then they use the Hanken Grotesk display family (600 weight, tight tracking).
2. Given body copy, when rendered, then it uses Inter at 16px / 1.5 line-height.
3. Given a scripture or reflection accent, when rendered, then it uses Newsreader italic and appears at most once per screen.


## L2-UX-004: Fixed BEM component set and 5R representations
**Traces to:** L1-UX-001
**Source:** implementation + mocks

The UI reuses a fixed BEM vocabulary (`.btn`, `.badge`, `.card`, `.rail`, `.gate`, `.dial`,
`.board`, `.col`, `.wcard`, `.movement`, `.pipstrip`, `.rlist`) and represents 5R progress three
consistent ways: the SVG dial (count), the vertical rlist (rail), and the horizontal pip strip
(cards).

**Acceptance Criteria:**
1. Given the components, when rendered, then they use the established BEM class names and match the mock markup structure.
2. Given a dial with N filled, when rendered, then N segments are lime, the (N+1)th is develop-green (current), the rest are the track color, and the center reads "N/5".
3. Given the dial, rlist, and pip strip for the same card, when rendered, then all three agree on which R's are done, current, and locked.


## L2-UX-005: Consistent locked-action visual language
**Traces to:** L1-UX-001
**Source:** implementation + mocks

Every gated action (advance phase, open Develop, mark done, drop into Done) uses the same locked
affordance: a padlock, dimmed styling, and `aria-disabled`.

**Acceptance Criteria:**
1. Given any gated action that is not yet permitted, when it renders, then it shows a padlock, `cursor: not-allowed`/dimmed styling, and `aria-disabled="true"`.
2. Given a gate whose requirements are all met, when re-evaluated, then the corresponding action is no longer locked.


## L2-UX-006: The Rhythm Rail signature element
**Traces to:** L1-UX-001
**Source:** implementation + mocks

The vertical 4D spine (the "Rhythm Rail") renders four station nodes with gate-latches between
them and, during Develop, expands into the 5R dial/list.

**Acceptance Criteria:**
1. Given a project screen, when the rail renders, then it shows the four ordered stations with their done/current/locked states and gate-latches (open/blocked/locked) between phases.
2. Given the Develop station is current, when the rail renders, then it nests the 5R dial and/or rlist for the active card.


## L2-UX-007: Responsive layout across viewports
**Traces to:** L1-UX-002
**Source:** implementation + mocks

The layout adapts at defined breakpoints; on narrow screens the rail becomes a horizontal top
strip and multi-column grids collapse.

**Acceptance Criteria:**
1. Given a viewport ≤ 760px, when a project screen renders, then the vertical rail flips to a horizontal top strip and the page does not scroll horizontally (`overflow-x: hidden`).
2. Given a viewport ≤ 900px, when grid sections render, then 3- and 4-column grids collapse to 2 columns.
3. Given viewports across XS (<576px), SM, MD, LG, and XL (≥1200px), when each screen renders, then content remains usable without horizontal overflow.


## L2-UX-008: Keyboard access and skip link
**Traces to:** L1-UX-002
**Source:** implementation + mocks

Every page provides a skip-to-content link and keyboard-operable controls with visible focus.

**Acceptance Criteria:**
1. Given any page, when it loads, then a "Skip to content" link targeting `#main` is the first focusable element.
2. Given keyboard navigation, when focus lands on an interactive control, then a visible focus outline is shown (develop-green focus color).


## L2-UX-009: Accessible enforcement controls
**Traces to:** L1-UX-002
**Source:** implementation

Interactive enforcement controls expose meaningful ARIA state.

**Acceptance Criteria:**
1. Given the 5R dial, when rendered, then it exposes an `aria-label` such as "3 of 5 movements complete: Request, Receive, Review".
2. Given a requirement toggle, when rendered, then it exposes `aria-pressed` and an actionable `aria-label`.
3. Given a locked action, when rendered, then it exposes `aria-disabled="true"`.
4. Given a client-side rejection (e.g. incomplete card dropped on Done), when it occurs, then the message is announced via `role="alert"`.


## L2-UX-010: Reduced-motion support
**Traces to:** L1-UX-002
**Source:** implementation + mocks

The UI honors `prefers-reduced-motion`.

**Acceptance Criteria:**
1. Given the user prefers reduced motion, when screens render, then animations and transitions are effectively disabled.


## L2-UX-011: Persistent dashboard navigation
**Traces to:** L1-UX-002
**Source:** implementation + mocks

Every authenticated screen provides a persistent route back to the workspace dashboard across
desktop and mobile layouts.

**Acceptance Criteria:**
1. Given any authenticated non-dashboard screen, when it renders, then its topbar includes a keyboard-operable Workspace link to `/dashboard`.
2. Given a project screen at a desktop viewport, when the Rhythm Rail renders, then its footer includes a Dashboard link.
3. Given a project screen at a viewport ≤ 760px, when the rail footer is hidden, then the topbar Workspace link remains visible and usable.
