# AUTH — Authentication: L2 Requirements

Detailed requirements for the [Authentication capability](./L1.md). Each requirement traces to one L1 in this capability. Acceptance criteria are in Given/When/Then form; acceptance test files must carry a `Traces to: L2-AUTH-###` comment header.

---

## L2-AUTH-001: Register a new account
**Traces to:** L1-AUTH-001
**Source:** implementation

`POST /api/auth/register` (anonymous) accepts `{ email, firstName, lastName, password }`,
creates a user with role `Member`, and returns an `AuthResult { accessToken, userId, email,
role, firstName, lastName, initials }`.

**Acceptance Criteria:**
1. Given valid registration details, when `POST /api/auth/register` is called, then the response is `200` and the body contains a non-empty `accessToken` and the submitted email, first name, and last name.
2. Given a successful registration, when the response is returned, then `role` is `"Member"` and `initials` is the uppercased first letters of first and last name (e.g. "Quinn"/"Brown" → `"QB"`).
3. Given first/last names with surrounding whitespace, when the account is created, then the stored names are trimmed.


## L2-AUTH-002: Registration password policy
**Traces to:** L1-AUTH-001
**Source:** implementation

Passwords must be at least 12 characters and contain an uppercase letter, a lowercase letter, a
digit, and a symbol.

**Acceptance Criteria:**
1. Given a password shorter than 12 characters, when registering, then the response is `400` with a validation error on `password`.
2. Given a 12+ character password missing an uppercase letter, a lowercase letter, a digit, or a symbol, when registering, then the response is `400` with the corresponding validation message ("must contain an uppercase letter" / "lowercase letter" / "a digit" / "a symbol").
3. Given `"Liturgy!2026"`, when registering, then the password passes validation.


## L2-AUTH-003: Registration field validation
**Traces to:** L1-AUTH-001
**Source:** implementation

Email must be present, well-formed, and ≤ 254 chars; first and last name must be present and
≤ 64 chars.

**Acceptance Criteria:**
1. Given a missing or malformed email, when registering, then the response is `400` with a validation error on `email`.
2. Given an email longer than 254 characters, when registering, then the response is `400`.
3. Given an empty first name or last name, or one longer than 64 characters, when registering, then the response is `400` with a validation error on that field.


## L2-AUTH-004: Duplicate email is rejected
**Traces to:** L1-AUTH-001
**Source:** implementation

Email is normalized (trimmed, lowercased) for storage and lookup; registering an
already-registered email is a conflict.

**Acceptance Criteria:**
1. Given an email already registered, when registering again with the same email (any casing/whitespace), then the response is `409` with title "Email already registered."
2. Given registration with a mixed-case email, when the user later signs in with a different casing of the same email, then sign-in succeeds.


## L2-AUTH-005: Sign in with valid credentials
**Traces to:** L1-AUTH-001
**Source:** implementation

`POST /api/auth/sign-in` (anonymous) accepts `{ email, password }` and returns an `AuthResult`
with a fresh access token.

**Acceptance Criteria:**
1. Given a registered user, when signing in with the correct email and password, then the response is `200` with a non-empty `accessToken` and the user's identity fields.
2. Given a missing email/password or malformed email, when signing in, then the response is `400`.


## L2-AUTH-006: Invalid credentials do not enumerate users
**Traces to:** L1-AUTH-001
**Source:** implementation

An unknown email and a wrong password produce the identical failure.

**Acceptance Criteria:**
1. Given an email that is not registered, when signing in, then the response is `401` with title "Invalid credentials."
2. Given a registered email with the wrong password, when signing in, then the response is `401` with the same "Invalid credentials." title and body as the unknown-email case.


## L2-AUTH-007: JWT access token issuance
**Traces to:** L1-AUTH-001
**Source:** implementation

Tokens are HMAC-SHA256 signed with the configured signing key and carry identity claims and an
expiry.

**Acceptance Criteria:**
1. Given a successful register or sign-in, when the token is issued, then it contains a `sub` claim = user id, an `email` claim, a role claim, and a unique `jti`.
2. Given default configuration, when a token is issued, then its expiry is 60 minutes after issuance (`Jwt:AccessTokenMinutes`, default 60).
3. Given a request bearing a token signed with a different key, when it reaches an authorized endpoint, then the response is `401`.


## L2-AUTH-008: Passwords hashed at rest
**Traces to:** L1-AUTH-001
**Source:** implementation

Passwords are stored only as BCrypt hashes (work factor 12); the plaintext is never persisted.

**Acceptance Criteria:**
1. Given a newly registered user, when the user row is inspected, then the stored `passwordHash` is a BCrypt hash and not the plaintext password.
2. Given the correct password, when verifying against the stored hash, then verification succeeds; given any other password, verification fails.


## L2-AUTH-009: Current-user endpoint
**Traces to:** L1-AUTH-001
**Source:** implementation

`GET /api/me` returns the authenticated user's profile; unauthenticated requests are rejected.

**Acceptance Criteria:**
1. Given a valid bearer token, when calling `GET /api/me`, then the response is `200` with `{ id, email, firstName, lastName, role, initials }` for that user.
2. Given no `Authorization` header, when calling `GET /api/me`, then the response is `401`.


## L2-AUTH-010: Initials derivation
**Traces to:** L1-AUTH-001
**Source:** implementation

Initials are the uppercased first character of the first and last name, falling back to `"?"`.

**Acceptance Criteria:**
1. Given first name "Quinn" and last name "Brown", when initials are derived, then the result is `"QB"`.
2. Given both names empty or whitespace, when initials are derived, then the result is `"?"`.


## L2-AUTH-011: Client auth session persistence
**Traces to:** L1-AUTH-001
**Source:** implementation

The Angular client stores the session under `localStorage['liturgy.auth']` as
`{ token, user }`, restores it on load, and degrades to in-memory when storage is unavailable.

**Acceptance Criteria:**
1. Given a successful sign-in, when the app reloads, then the user remains authenticated (session restored from `localStorage`).
2. Given corrupt or unparsable stored session data, when the app initializes, then the stored value is removed and the app treats the user as signed out (no crash).
3. Given `signOut()`, when invoked, then the stored session is cleared and the user is redirected to `/sign-in`.


## L2-AUTH-012: Route auth guard
**Traces to:** L1-AUTH-001
**Source:** implementation

`authGuard` protects all in-shell routes; unauthenticated navigation redirects to sign-in.

**Acceptance Criteria:**
1. Given no active session, when navigating to `/projects` (or any shell route), then the app redirects to `/sign-in`.
2. Given an active session, when navigating to a shell route, then the route activates.


## L2-AUTH-013: Bearer token attached to API calls
**Traces to:** L1-AUTH-001
**Source:** implementation

`authInterceptor` adds `Authorization: Bearer <token>` to outgoing HTTP requests when a token is
present.

**Acceptance Criteria:**
1. Given an authenticated session, when any `@liturgy/api` service issues an HTTP request, then the request carries `Authorization: Bearer <token>`.
2. Given no token, when a request is issued, then it is sent without an `Authorization` header.


## L2-AUTH-014: Sign-in and sign-up screens
**Traces to:** L1-AUTH-001
**Source:** implementation + mocks

`/sign-in` and `/sign-up` present reactive forms with client validation and human error copy;
success navigates to `/projects`.

**Acceptance Criteria:**
1. Given valid credentials entered on `/sign-in`, when the form is submitted, then the app stores the session and navigates to `/projects`.
2. Given a failed sign-in, when the server rejects it, then the UI shows "That email and password did not match. Please try again." and stays on `/sign-in`.
3. Given the sign-up form, when the email is registered already (`409`), then the UI shows "That email is already registered. Try signing in."; on other validation failure it shows the password-policy message.
4. Given the sign-up password field, when validated client-side, then it requires at least 12 characters.
