# L1 — High-Level Requirements

Reverse-engineered from the `docs/mocks` design set and the implemented full-stack vertical
slice (`backend/` .NET 9 Clean Architecture; `frontend/` Angular 21). Liturgy is an agile
workspace that **enforces the FaithTech Playbook**: projects advance through the **4D Cycle**
(Discover → Discern → Develop → Demonstrate) behind requirement gates, and every Develop work
item must complete the **5R loop** (Request → Receive → Review → Render → Rejoice) before it
can reach Done. Enforcement is server-authoritative; collaborative surfaces update in real time.

Each L2 records a **Source** tag: `implementation` (built and tested in the slice),
`implementation + mocks` (built, and also specified visually), or `mocks (deferred)` (present
in the design mocks, deferred to a follow-up per the implementation plan). Mock-derived
requirements are stated as "must" because they define the intended product; teams building the
deferred surfaces trace to them.

---

## L1-001: User Authentication & Identity
The system must let a person register an account and sign in with email and password, must
issue a signed access token establishing their identity, and must expose the currently
authenticated user's profile (including derived initials) to authorized callers.

## L1-002: Workspace Membership & Roles
The system must organize users into a workspace with named members, each carrying a workspace
role and display initials, so that projects, boards, and cards can be attributed to people.

## L1-003: Project Listing & Workspace Home
The system must present the workspace's projects — each with its name, tag, and current 4D
phase — as a browsable list, and must surface workspace-level momentum and the gates that need
attention on a dashboard home.

## L1-004: 4D Cycle Project Journey
The system must render a project's journey through the four sequential 4D phases as an ordered
spine, showing each phase's state (Locked / Current / Done) and its gate, so a team can see
exactly where the project stands and what must be completed to advance.

## L1-005: Enforcement Gates & Phase Unlocking
The system must gate each phase transition behind a checklist of requirements. A gate must stay
Blocked until every requirement on it is Done, must open automatically the moment the last
requirement is completed, and — on opening — must mark its phase Done and unlock the next phase.
Gate and phase state must be decided by the server, never the client.

## L1-006: Discern Decision Gate
The system must record a project's Discern decision as exactly one of four discernment paths
(Reject · Receive · Reimagine · Create) together with a written rationale, and must not permit
Develop to open until the decision and its gate requirements are complete.

## L1-007: Develop Kanban Board
The system must present the Develop phase as a Kanban board with Backlog, In-the-5R-loop,
Review, and Done columns; must let members create, assign, and move cards; and must display each
card's 5R progress. The Done column must be locked so that no card can enter it until its 5R
loop is complete.

## L1-008: 5R Co-Creation Loop
The system must run the five-movement 5R loop on every work item, must require the movements to
be logged strictly in order, must capture the reflective content of each movement, and must not
permit a card to be marked Done until all five movements are logged.

## L1-009: Server-Authoritative Enforcement Engine
The system must contain a pure, isolable enforcement core that computes all derived enforcement
state — movement states, a card's current R, whether a column may be entered, and whether a
gate is open — from a single source of truth (which movements/requirements are complete), so the
rules are cheaply and deterministically testable.

## L1-010: Real-Time Collaboration
The system must broadcast state changes (card moves, card creation/assignment, movement logs,
requirement toggles, gate changes, phase unlocks) to all members viewing the same project, so
that concurrent collaborators see an up-to-date board, loop, and journey without manual refresh.
The application must degrade cleanly to request/response when a live channel is unavailable.

## L1-011: Demonstrate & Impact
The system must present the Demonstrate phase as an impact surface that reframes success as
"friendship compounded by time" — showing relationship-oriented metrics, a longitudinal story
timeline, and a "Rejoice" gratitude wall — rather than shipping speed.

## L1-012: Design System & Visual Language
The system must implement a single, documented design system — FaithTech's color tokens, phase
color coding, three-typeface hierarchy, and a fixed set of BEM components — and must render the
signature elements (the Rhythm Rail, the 5R dial, pip strips, gates, work cards) consistently
across every screen.

## L1-013: Responsive & Accessible UI
The system's web UI must be usable across mobile, tablet, and desktop viewports and must meet
baseline accessibility: keyboard operability, skip-to-content, meaningful ARIA state on
interactive enforcement controls, visible focus, and reduced-motion support.

## L1-014: Security & Data Protection
The system must require authentication for every data endpoint except registration and sign-in,
must validate all input server-side, must protect credentials (hashed at rest, never returned or
logged), must not leak whether an email exists, and must restrict browser cross-origin access to
the approved application origin.

## L1-015: API Contract, Error Handling & Observability
The system must expose a consistent HTTP/JSON contract: enum values serialized as strings,
successful commands returning the updated resource, and every error returned as a
`problem+json` response with a status code that correctly classifies the failure
(validation → 400, auth → 401, conflict/enforcement → 409, missing → 404, unexpected → 500).

## L1-016: Performance & Reliability
The system must respond to interactive operations quickly, keep the enforcement core free of I/O
so it evaluates in-memory, apply database migrations and seed demo data deterministically on
startup, and expose a health endpoint for readiness checks.
