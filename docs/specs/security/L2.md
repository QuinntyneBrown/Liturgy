# SEC — Security: L2 Requirements

Detailed requirements for the [Security capability](./L1.md). Each requirement traces to one L1 in this capability. Acceptance criteria are in Given/When/Then form; acceptance test files must carry a `Traces to: L2-SEC-###` comment header.

---

## L2-SEC-001: Authentication required on all data endpoints
**Traces to:** L1-SEC-001
**Source:** implementation

Only registration and sign-in are anonymous; every other endpoint requires a valid bearer token.

**Acceptance Criteria:**
1. Given no bearer token, when calling `/api/me`, `/api/projects`, `/api/board/**`, `/api/loop/**`, or `/api/gates/**`, then the response is `401`.
2. Given a valid bearer token, when calling those endpoints, then the request is authorized.


## L2-SEC-002: Server-side input validation
**Traces to:** L1-SEC-001
**Source:** implementation

All command input is validated server-side via the validation pipeline before any handler runs;
failures return structured field errors.

**Acceptance Criteria:**
1. Given input violating a validation rule, when the command is dispatched, then the handler does not execute and the response is `400` with a `ValidationProblemDetails` body listing per-field errors.
2. Given the client also enforces a rule (e.g. password length), when the client is bypassed, then the server still rejects the invalid input.


## L2-SEC-003: No user enumeration; credentials protected
**Traces to:** L1-SEC-001
**Source:** implementation

Authentication failures are indistinguishable across unknown-email and wrong-password, and
password material is never exposed.

**Acceptance Criteria:**
1. Given sign-in failures, when they occur, then unknown-email and wrong-password return an identical `401` response.
2. Given any API response or log, when inspected, then no plaintext password or password hash is present in responses.


## L2-SEC-004: Cross-origin access restricted to the app origin
**Traces to:** L1-SEC-001
**Source:** implementation

CORS allows the approved application origin (`http://localhost:4200`) with credentials, as
required for the authenticated SignalR channel.

**Acceptance Criteria:**
1. Given a browser request from `http://localhost:4200`, when it hits the API or hub, then CORS permits it with credentials allowed.
2. Given a request from a non-approved origin, when it is made from a browser, then it is not granted cross-origin access.
