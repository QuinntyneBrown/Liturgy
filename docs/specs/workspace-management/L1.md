# WKSP — Workspace Management

High-level (L1) requirements for the **Workspace Management** capability. Detailed L2 requirements live in [L2.md](./L2.md).

> **Account = Workspace.** The product surfaces the root tenant as the "Account"; in the domain
> it is the `Workspace` entity, which is the canonical name used throughout these specs.

---

## L1-WKSP-001: Workspace Membership & Roles
*(formerly L1-002)*

The system must organize users into a workspace with named members, each carrying a workspace
role and display initials, so that projects, boards, and cards can be attributed to people.

## L1-WKSP-002: Project Listing & Workspace Home
*(formerly L1-003)*

The system must present the workspace's projects — each with its name, tag, and current 4D
phase — as a browsable list, and must surface workspace-level momentum and the gates that need
attention on a dashboard home. It must also let a member manage a project's lifecycle: rename or
re-tag it, close it (soft-hide, so it drops out of the default list) and reopen it, or hard
delete it and all its children. Each project carries a `Status` (`ProjectStatus {Active,
Closed}`, default `Active`).

## L1-WKSP-003: Account Membership by Invitation

The system must let a workspace **Lead** grow the account by inviting people by email. An
invitation is created in-app and carries a token and an `InvitationStatus {Pending, Accepted,
Revoked}`; there is no email delivery — the Lead shares the returned invite URL out of band.
An existing user accepts an invitation to join the inviting account; a new user joins by
registering with the invitation's token (see [L2-AUTH-016](../authentication/L2.md#l2-auth-016-registration-with-an-invitation-token-joins-the-inviting-account)).
A Lead may list pending invitations and revoke any that are still pending.
