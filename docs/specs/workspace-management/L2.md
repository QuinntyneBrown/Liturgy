# WKSP — Workspace Management: L2 Requirements

Detailed requirements for the [Workspace Management capability](./L1.md). Each requirement traces to one L1 in this capability. Acceptance criteria are in Given/When/Then form; acceptance test files must carry a `Traces to: L2-WKSP-###` comment header.

---

## L2-WKSP-001: Workspace with named members
**Traces to:** L1-WKSP-001
**Source:** implementation + mocks

A workspace groups members; each membership carries a workspace role and initials. The demo
workspace is "New Hope Collective" with members Quinn Brown (QB, Lead), Amara Mensah (AM,
Designer), Jonah Park (JP, Developer), and Sam Doyle (SD, Developer).

**Acceptance Criteria:**
1. Given the seeded workspace, when it is loaded, then it is named "New Hope Collective" (slug `new-hope-collective`) and has four members with initials QB, AM, JP, SD.
2. Given a member, when their card assignment is displayed, then the card shows that member's initials avatar.


## L2-WKSP-002: Assign a card to a member
**Traces to:** L1-WKSP-001
**Source:** implementation

`POST /api/board/cards/{cardId}/assign` sets or clears a card's assignee.

**Acceptance Criteria:**
1. Given an existing card and a member id, when assign is called, then the card's `assigneeId` and `assigneeInitials` reflect that member and a `CardAssigned` event is broadcast.
2. Given a null assignee id, when assign is called, then the card becomes unassigned.
3. Given a non-existent card id, when assign is called, then the response is `404`.


## L2-WKSP-003: List projects
**Traces to:** L1-WKSP-002
**Source:** implementation

`GET /api/projects` returns every project with `{ id, name, tag, currentPhase }`, ordered by
creation time.

**Acceptance Criteria:**
1. Given the workspace has projects, when `GET /api/projects` is called with a valid token, then the response is `200` with a list of `{ id, name, tag, currentPhase }` ordered by `createdAt`.
2. Given no token, when `GET /api/projects` is called, then the response is `401`.


## L2-WKSP-004: Projects landing page
**Traces to:** L1-WKSP-002
**Source:** implementation + mocks

`/projects` renders a tile per project showing name, tag, and current-phase badge; a tile opens
that project's journey.

**Acceptance Criteria:**
1. Given the seeded data, when the user opens `/projects`, then a tile labeled "Lantern" is visible.
2. Given a project tile, when it is activated, then the app navigates to `/projects/{id}`.


## L2-WKSP-005: Workspace dashboard momentum and attention feed
**Traces to:** L1-WKSP-002
**Source:** mocks (deferred)

The dashboard home shows momentum stats (active projects, 5R movements logged this week, gates
blocked, weeks of friendship compounded), surfaces blocked gates in a "Gates that need
attention" panel, and plots every project on a four-lane 4D board.

**Acceptance Criteria:**
1. Given the workspace, when the dashboard loads, then it shows a stat tile for active-project count and a "Gates blocked — need attention" count styled with the alert (discern-red) treatment.
2. Given one or more blocked gates, when the dashboard loads, then each is listed in "Gates that need attention" with the project name, the reason it is blocked, and an action link to open the gate or 5R loop.
3. Given all projects, when the dashboard loads, then each appears in the lane matching its current 4D phase (Discover / Discern / Develop / Demonstrate) with a phase-colored badge.


## L2-WKSP-006: Update a project
**Traces to:** L1-WKSP-002
**Source:** implementation

`PUT /api/projects/{id}` updates a project's `name` and `tag`.

**Acceptance Criteria:**
1. Given an existing project, when `PUT /api/projects/{id}` is called with a new name and tag, then the response is `2xx` and a subsequent read reflects the updated name and tag.
2. Given a missing or over-length name or tag, when updating, then the response is `400`.
3. Given a non-existent project id (or a project the caller is not a member of), when updating, then the response is `404`.


## L2-WKSP-007: Close a project
**Traces to:** L1-WKSP-002
**Source:** implementation

`POST /api/projects/{id}/close` sets the project's `Status` to `Closed`; a closed project is
soft-hidden — retained with all its data but excluded from the default project list.

**Acceptance Criteria:**
1. Given an `Active` project, when `POST /api/projects/{id}/close` is called, then the response is `2xx` and the project's `Status` becomes `Closed`.
2. Given a project just closed, when `GET /api/projects` is called without `includeClosed`, then the project is absent from the list (see L2-WKSP-010).
3. Given a non-existent project id (or one the caller is not a member of), when closing, then the response is `404`.


## L2-WKSP-008: Reopen a project
**Traces to:** L1-WKSP-002
**Source:** implementation

`POST /api/projects/{id}/reopen` sets a `Closed` project's `Status` back to `Active`, restoring
it to the default list.

**Acceptance Criteria:**
1. Given a `Closed` project, when `POST /api/projects/{id}/reopen` is called, then the response is `2xx` and the project's `Status` becomes `Active`.
2. Given a project just reopened, when `GET /api/projects` is called, then the project appears in the default list again.
3. Given a non-existent project id (or one the caller is not a member of), when reopening, then the response is `404`.


## L2-WKSP-009: Delete a project
**Traces to:** L1-WKSP-002
**Source:** implementation

`DELETE /api/projects/{id}` hard-deletes the project and all of its children (phases, gates,
requirements, sprints, cards, and 5R movements). Unlike Close, deletion is irreversible.

**Acceptance Criteria:**
1. Given an existing project, when `DELETE /api/projects/{id}` is called, then the response is `2xx` and a subsequent `GET /api/projects/{id}` returns `404`.
2. Given a deleted project, when its board is requested, then the project's cards and their movements are gone (children are removed with the project).
3. Given a non-existent project id (or one the caller is not a member of), when deleting, then the response is `404`.


## L2-WKSP-010: Default project list hides Closed projects
**Traces to:** L1-WKSP-002
**Source:** implementation

`GET /api/projects` returns only `Active` projects by default; an `includeClosed` option reveals
`Closed` ones as well. This refines [L2-WKSP-003](#l2-wksp-003-list-projects).

**Acceptance Criteria:**
1. Given the account has both `Active` and `Closed` projects, when `GET /api/projects` is called without `includeClosed`, then only the `Active` projects are returned.
2. Given the same account, when `GET /api/projects` is called with `includeClosed` set, then both `Active` and `Closed` projects are returned.


## L2-WKSP-011: Invite a member by email
**Traces to:** L1-WKSP-003
**Source:** implementation

`POST /api/invitations` lets a workspace `Lead` invite a person by email with an optional role
(default `Member`). It creates a `Pending` `Invitation` bearing a token and returns the token
and an invite URL.

**Acceptance Criteria:**
1. Given a caller who is a `Lead` in the account, when `POST /api/invitations` is called with an email and optional role, then the response is `2xx` and the body contains a `token` and an invite URL, and a `Pending` invitation is recorded for that email and role.
2. Given a caller who is a member but not a `Lead`, when inviting, then the response is `403`.
3. Given a missing or malformed email, when inviting, then the response is `400`.


## L2-WKSP-012: List pending invitations
**Traces to:** L1-WKSP-003
**Source:** implementation

`GET /api/invitations` lists the caller's account's `Pending` invitations.

**Acceptance Criteria:**
1. Given the account has pending invitations, when a `Lead` calls `GET /api/invitations`, then the response is `2xx` with those invitations (email, role, status `Pending`), scoped to the caller's account.
2. Given an invitation that has been `Accepted` or `Revoked`, when the list is read, then it is not included among the pending invitations.
3. Given no token, when `GET /api/invitations` is called, then the response is `401`.


## L2-WKSP-013: Anonymous invite context lookup
**Traces to:** L1-WKSP-003
**Source:** implementation

`GET /api/invitations/{token}` is **anonymous** and returns `{ workspaceName, invitedByName,
email }` so the sign-up screen can show who invited the person and to which account.

**Acceptance Criteria:**
1. Given a `Pending` invitation, when `GET /api/invitations/{token}` is called without authentication, then the response is `200` with `{ workspaceName, invitedByName, email }`.
2. Given a token that is missing, already `Accepted`, or `Revoked`, when the endpoint is called, then the response is `404`.


## L2-WKSP-014: Accept an invitation
**Traces to:** L1-WKSP-003
**Source:** implementation

`POST /api/invitations/{token}/accept` lets an **authenticated** existing user join the inviting
account: it creates a `Membership` in that account with the invitation's role and marks the
invitation `Accepted`. The authenticated user's email must equal the invitation's email.

**Acceptance Criteria:**
1. Given an authenticated user whose email equals the invitation's email and a `Pending` invitation, when `POST /api/invitations/{token}/accept` is called, then a `Membership` in the inviting account is created with the invitation's role, the invitation becomes `Accepted`, and the response is `2xx`.
2. Given an authenticated user whose email does **not** equal the invitation's email, when accepting, then the response is `400` (validation), no membership is created, and the invitation stays `Pending`.
3. Given the accepting user is already a member of the inviting account, when accepting, then no duplicate membership is created.
4. Given a token that is missing, already `Accepted`, or `Revoked`, when accepting, then the response is `404`; given no authentication, then the response is `401`.


## L2-WKSP-015: Revoke a pending invitation
**Traces to:** L1-WKSP-003
**Source:** implementation

`DELETE /api/invitations/{id}` lets a `Lead` revoke a `Pending` invitation, setting its status
to `Revoked` so its token can no longer be accepted.

**Acceptance Criteria:**
1. Given a `Pending` invitation, when a `Lead` calls `DELETE /api/invitations/{id}`, then the response is `2xx` and the invitation's status becomes `Revoked`.
2. Given a revoked invitation, when its token is looked up (L2-WKSP-013) or accepted (L2-WKSP-014), then the response is `404`.
3. Given a caller who is a member but not a `Lead` in the account, when revoking, then the response is `403`; given an invitation that is missing or belongs to another account, then the response is `404` (the two `404` cases are intentionally indistinguishable).
4. Given an invitation that is not `Pending` (already `Accepted` or `Revoked`), when revoking, then the response is `409`.


## L2-WKSP-016: Registration with an invitation token joins the inviting account
**Traces to:** L1-WKSP-003
**Source:** implementation

A new user may join an account by registering with the invitation's token instead of accepting
as an existing user. The registration-side contract and its criteria are specified in
[L2-AUTH-016](../authentication/L2.md#l2-auth-016-registration-with-an-invitation-token-joins-the-inviting-account);
this requirement records the membership outcome as part of the invitation capability.

**Acceptance Criteria:**
1. Given a `Pending` invitation whose email matches a new registrant, when they register with its token, then a `Membership` in the inviting account is created with the invitation's role and the invitation becomes `Accepted` (no personal account is created).
2. Given the same, when the registrant's email does not match the invitation, then the invitation is ignored and remains `Pending`.
