# PLAY — Playbook Enforcement: L2 Requirements

Detailed requirements for the [Playbook Enforcement capability](./L1.md). Each requirement traces to one L1 in this capability. Acceptance criteria are in Given/When/Then form; acceptance test files must carry a `Traces to: L2-PLAY-###` comment header.

---

## L2-PLAY-001: Retrieve a project's 4D journey
**Traces to:** L1-PLAY-001
**Source:** implementation

`GET /api/projects/{id}` returns the project plus its four phases in order, each with its state
and (where present) its gate and requirements.

**Acceptance Criteria:**
1. Given a project id, when `GET /api/projects/{id}` is called, then the response contains `phases` ordered by `order`, each with `{ id, kind, state, order, gate }`.
2. Given a phase with a gate, when the journey is returned, then the gate includes `{ id, phaseId, title, state, requirements[] }` and each requirement includes `{ id, label, meta, state, order }`.
3. Given the seeded Lantern project, when the journey is returned, then Discover and Discern are `Done`, Develop is `Current`, and Demonstrate is `Locked`.


## L2-PLAY-002: Journey spine rendering
**Traces to:** L1-PLAY-001
**Source:** implementation + mocks

`/projects/:id` renders the phases as a connected vertical spine; each node shows a checkmark
when Done or its ordinal otherwise, and carries `is-done` / `is-current` / `is-locked` styling
and a phase-colored state badge.

**Acceptance Criteria:**
1. Given a Done phase, when its row renders, then the node shows a checkmark and the row carries the `is-done` class.
2. Given the Current phase, when its row renders, then it carries `is-current` and its badge shows the current state text.
3. Given a Locked phase, when its row renders, then it carries `is-locked` and reads "Locked".


## L2-PLAY-003: Open board from the Develop phase
**Traces to:** L1-PLAY-001
**Source:** implementation

When the Develop phase is not Locked, the journey offers a link into that project's board.

**Acceptance Criteria:**
1. Given Develop is `Current` or `Done`, when the journey renders, then an "Open board →" link to `/board/{projectId}` is shown on the Develop row.
2. Given Develop is `Locked`, when the journey renders, then no board link is shown on the Develop row.


## L2-PLAY-004: Toggle a gate requirement
**Traces to:** L1-PLAY-002
**Source:** implementation

`POST /api/gates/requirements/{requirementId}/toggle` with `{ done }` sets the requirement to
Done or Todo and returns the recomputed gate.

**Acceptance Criteria:**
1. Given a requirement, when toggled with `{ done: true }`, then its state becomes `Done` and the returned gate reflects the recomputed state.
2. Given a requirement, when toggled with `{ done: false }`, then its state becomes `Todo`.
3. Given a non-existent requirement id, when toggling, then the response is `404` (title "Requirement not found.").


## L2-PLAY-005: Gate opens only when all requirements are Done
**Traces to:** L1-PLAY-002
**Source:** implementation

The gate state is computed by the server from its requirements — Open iff every requirement is
Done, otherwise Blocked. The client never sets gate state.

**Acceptance Criteria:**
1. Given a gate with any requirement still `Todo`, when its state is evaluated, then the gate is `Blocked`.
2. Given a gate whose last remaining `Todo` requirement is toggled Done, when the toggle completes, then the returned gate state is `Open`.
3. Given a gate that is Open, when a requirement is toggled back to `Todo`, then the gate returns to `Blocked`.


## L2-PLAY-006: Opening a gate unlocks the next phase
**Traces to:** L1-PLAY-002
**Source:** implementation

When a gate transitions Blocked→Open, its phase becomes Done, the next phase (order+1) becomes
Current, and the project's `currentPhase` advances to that next phase.

**Acceptance Criteria:**
1. Given the seeded Lantern gate one requirement short of Open, when the last requirement is toggled Done, then the gate's phase becomes `Done`, the following phase becomes `Current`, and `project.currentPhase` equals the next phase's kind.
2. Given a gate that is already Open, when another requirement on it is toggled, then no phase transition occurs a second time (the unlock fires only on the Blocked→Open transition).
3. Given a phase with no following phase (the last phase), when its gate opens, then the phase is marked Done and no unlock cascade error occurs.


## L2-PLAY-007: Blocked gate presentation and locked advance action
**Traces to:** L1-PLAY-002
**Source:** implementation + mocks

A blocked gate renders with a padlock, a "N left" blocked badge counting remaining requirements,
and a disabled advance button; an open gate renders a check and an "Open" done badge.

**Acceptance Criteria:**
1. Given a blocked gate with N `Todo` requirements, when it renders, then it shows the `badge--blocked` state reading the remaining count and a locked/disabled advance control (`aria-disabled`, padlock).
2. Given an open gate, when it renders, then it shows `badge--done` reading "Open" and a check icon.
3. Given the journey after a client toggles the last requirement, when the client re-fetches the journey, then the gate and phase states reflect the server's recomputed result (client does not compute them locally).


## L2-PLAY-008: Requirement checkbox is an accessible toggle
**Traces to:** L1-PLAY-002
**Source:** implementation

Each requirement renders as a button exposing its pressed state and an actionable label.

**Acceptance Criteria:**
1. Given a requirement row, when rendered, then it is a `button` with `aria-pressed` reflecting Done state and `aria-label` "Toggle: {label}".
2. Given a Done requirement, when rendered, then its checkbox shows the completed (checked) styling; given a Todo requirement, it shows the empty outlined styling.


## L2-PLAY-009: Gate and phase changes broadcast in real time
**Traces to:** L1-PLAY-002
**Source:** implementation

Toggling always emits `RequirementToggled`; a state change additionally emits `GateChanged`; an
unlock additionally emits `PhaseUnlocked`.

**Acceptance Criteria:**
1. Given a project group with subscribers, when a requirement is toggled, then a `RequirementToggled` event carrying the gate is sent to `project:{id}`.
2. Given a toggle that changes the gate's state, when it completes, then a `GateChanged` event is also sent.
3. Given a toggle that unlocks the next phase, when it completes, then a `PhaseUnlocked` event carrying the next phase (with its gate) is also sent, and a second browser viewing the journey updates without manual refresh.


## L2-PLAY-010: Choose exactly one discernment path
**Traces to:** L1-PLAY-003
**Source:** mocks (deferred)

The Discern screen offers four mutually exclusive paths — Reject, Receive, Reimagine, Create —
of which exactly one may be chosen and recorded.

**Acceptance Criteria:**
1. Given the four path choices, when one is selected, then it is marked chosen (`aria-pressed="true"`, chosen styling) and the others are not.
2. Given a chosen path, when a different path is selected, then the selection moves so that exactly one path is chosen at any time.
3. Given the seeded Wellspring project, when the Discern screen loads, then "Reimagine" is the recorded chosen path.


## L2-PLAY-011: Record the discernment rationale
**Traces to:** L1-PLAY-003
**Source:** mocks (deferred)

The decision captures a written "Why this path?" rationale and who it was prayed over with.

**Acceptance Criteria:**
1. Given a chosen path, when the rationale is written and saved, then the note text and the "Prayed over with" value are persisted with the decision.
2. Given a Discern gate requirement "Path chosen", when a path is recorded, then that requirement is satisfied.


## L2-PLAY-012: Develop stays locked until Discern is complete
**Traces to:** L1-PLAY-003
**Source:** mocks (deferred)

The Discern→Develop gate remains Blocked (and the "Open Develop" action locked) until all its
requirements — including a defined sprint goal — are Done.

**Acceptance Criteria:**
1. Given the Discern→Develop gate with the "Sprint goal defined" requirement still Todo, when the screen renders, then the "Open Develop" button is locked (padlock, `aria-disabled`) and the gate reads the remaining count.
2. Given every Discern gate requirement Done, when the gate is evaluated, then it opens and Develop unlocks per L2-PLAY-006.


## L2-PLAY-013: All-movements-logged rule
**Traces to:** L1-PLAY-004
**Source:** implementation

`AllMovementsLogged` is true only when the count of movements with a set `loggedAt` reaches five.

**Acceptance Criteria:**
1. Given 0, 3, or 4 movements logged, when evaluated, then `AllMovementsLogged` is false.
2. Given 5 movements logged, when evaluated, then `AllMovementsLogged` is true.


## L2-PLAY-014: Next-R computation
**Traces to:** L1-PLAY-004
**Source:** implementation

`NextR` returns the first R in loop order that is not yet logged, or null when all are logged.

**Acceptance Criteria:**
1. Given 0 logged, when computing next R, then the result is `Request`.
2. Given 3 logged, when computing, then the result is `Render`; given 4 logged, then `Rejoice`.
3. Given 5 logged, when computing, then the result is null.


## L2-PLAY-015: Movement-state recomputation
**Traces to:** L1-PLAY-004
**Source:** implementation

`RecomputeMovementStates` sets logged movements Done, the first unlogged movement Current, and
the remainder Locked, in `order`.

**Acceptance Criteria:**
1. Given 2 movements logged, when states are recomputed, then the sequence is [Done, Done, Current, Locked, Locked].
2. Given 5 logged, when recomputed, then all are Done and none is Current.


## L2-PLAY-016: Column-entry rule
**Traces to:** L1-PLAY-004
**Source:** implementation

`CanEnterColumn` permits any target except Done, which requires a complete loop.

**Acceptance Criteria:**
1. Given any movement count, when the target is `Backlog`, `InLoop`, or `Review`, then entry is allowed.
2. Given 4 logged, when the target is `Done`, then entry is denied; given 5 logged, then entry is allowed.


## L2-PLAY-017: Gate-evaluation rule
**Traces to:** L1-PLAY-004
**Source:** implementation

`EvaluateGate` returns Open iff every requirement is Done (an empty requirement set is Open),
else Blocked.

**Acceptance Criteria:**
1. Given any requirement in `Todo`, when the gate is evaluated, then it is `Blocked`.
2. Given all requirements `Done`, when evaluated, then the gate is `Open`.
3. Given an empty requirement set, when evaluated, then the gate is `Open`.


## L2-PLAY-018: Enforcement core is pure and I/O-free
**Traces to:** L1-PLAY-004
**Source:** implementation

The engine performs no database or network I/O; every rule is unit-testable in memory.

**Acceptance Criteria:**
1. Given the engine, when its rules are exercised in unit tests, then no database, HTTP, or SignalR dependency is required.
2. Given identical inputs, when a rule is evaluated repeatedly, then the output is deterministic.
