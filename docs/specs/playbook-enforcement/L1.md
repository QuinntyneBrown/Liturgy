## L1-PLAY-001: 4D Cycle Project Journey
*(formerly L1-004)*

The system must render a project's journey through the four sequential 4D phases as an ordered
spine, showing each phase's state (Locked / Current / Done) and its gate, so a team can see
exactly where the project stands and what must be completed to advance.

## L1-PLAY-002: Enforcement Gates & Phase Unlocking
*(formerly L1-005)*

The system must gate each phase transition behind a checklist of requirements. A gate must stay
Blocked until every requirement on it is Done, must open automatically the moment the last
requirement is completed, and — on opening — must mark its phase Done and unlock the next phase.
Gate and phase state must be decided by the server, never the client.

## L1-PLAY-003: Discern Decision Gate
*(formerly L1-006)*

The system must record a project's Discern decision as exactly one of four discernment paths
(Reject · Receive · Reimagine · Create) together with a written rationale, and must not permit
Develop to open until the decision and its gate requirements are complete.

## L1-PLAY-004: Server-Authoritative Enforcement Engine
*(formerly L1-009)*

The system must contain a pure, isolable enforcement core that computes all derived enforcement
state — movement states, a card's current R, whether a column may be entered, and whether a
gate is open — from a single source of truth (which movements/requirements are complete), so the
rules are cheaply and deterministically testable.
