# COLLAB — Collaboration: L2 Requirements

Detailed requirements for the [Collaboration capability](./L1.md). Each requirement traces to one L1 in this capability. Acceptance criteria are in Given/When/Then form; acceptance test files must carry a `Traces to: L2-COLLAB-###` comment header.

---

## L2-COLLAB-001: Authorized project-scoped hub
**Traces to:** L1-COLLAB-001
**Source:** implementation

The SignalR hub at `/hubs/board` requires authentication and lets clients join/leave a
per-project group.

**Acceptance Criteria:**
1. Given an unauthenticated connection, when it attempts to connect to `/hubs/board`, then the connection is rejected (hub carries `[Authorize]`).
2. Given an authenticated client, when it invokes `JoinProject(projectId)`, then it is added to group `project:{projectId}`; `LeaveProject(projectId)` removes it.


## L2-COLLAB-002: Server broadcasts scoped to the project group
**Traces to:** L1-COLLAB-001
**Source:** implementation

Each enforcement mutation notifies only the affected project's group with a typed event and
payload.

**Acceptance Criteria:**
1. Given a mutation on project P, when it completes, then the corresponding event (`CardMoved`/`CardCreated`/`CardAssigned` → CardDto, `MovementLogged` → CardLoopDto, `RequirementToggled`/`GateChanged` → GateDto, `PhaseUnlocked` → PhaseDto) is sent only to `project:{P}`.
2. Given a client joined to project P, when a mutation happens on a different project, then that client receives no event.


## L2-COLLAB-003: Client subscribes and reconciles live
**Traces to:** L1-COLLAB-001
**Source:** implementation

The client connects with the access token, joins the project on entry, rejoins on reconnect, and
reloads the affected view on each event.

**Acceptance Criteria:**
1. Given the board is open, when a `CardMoved`/`CardCreated`/`CardAssigned` event arrives, then the board reloads to reflect it.
2. Given the loop is open for card C, when a `MovementLogged` event for C arrives, then the loop updates; an event for another card is ignored.
3. Given the connection drops and reconnects, when it is restored, then the client re-invokes `JoinProject` for the current project.


## L2-COLLAB-004: Degrade cleanly without a live channel
**Traces to:** L1-COLLAB-001
**Source:** implementation

When the realtime channel is unavailable, the app still functions over REST.

**Acceptance Criteria:**
1. Given the hub connection cannot be established (e.g. `/hubs/**` blocked), when the user drives the app, then all screens still load and mutate correctly via REST (no crash, no blocking spinner).
2. Given a test/no-op realtime provider, when injected, then the app behaves identically minus live push updates.
