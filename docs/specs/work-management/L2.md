Detailed requirements for the [Work Management capability](./L1.md). Each requirement traces to one L1 in this capability. Acceptance criteria are in Given/When/Then form; acceptance test files must carry a `Traces to: L2-WORK-###` comment header.

## L2-WORK-001: Retrieve the board
**Traces to:** L1-WORK-001
**Source:** implementation

`GET /api/board/{projectId}` returns the current sprint and its cards, each with 5R progress.

**Acceptance Criteria:**
1. Given a project, when `GET /api/board/{projectId}` is called, then the response contains `{ projectId, sprintId, sprintNumber, cards[] }`.
2. Given each card, when the board is returned, then the card includes `{ id, code, title, assigneeId, assigneeInitials, column, currentR, isBlocked, loggedCount }`.
3. Given the seeded Lantern board, when returned, then it is Sprint 6 and includes cards LAN-24 (InLoop), LAN-33 (Backlog), and LAN-12 (Done).


## L2-WORK-002: Create a card
**Traces to:** L1-WORK-001
**Source:** implementation

`POST /api/board/cards` with `{ projectId, title, assigneeId? }` creates a card in Backlog with
a derived code and a fresh, unlogged 5R loop.

**Acceptance Criteria:**
1. Given a valid project and title, when a card is created, then it starts in `Backlog` with `currentR = Request`, `loggedCount = 0`, five movements all initialized (Request `Current`, the rest `Locked`), and a `CardCreated` event is broadcast.
2. Given project "Lantern" with highest existing code LAN-33, when a card is created, then its code is `LAN-34` (prefix = first â‰¤3 letters uppercased, number = max suffix + 1).
3. Given a missing title or a title longer than 200 characters, when creating, then the response is `400`.
4. Given a non-existent project id, when creating, then the response is `404` (title "Project not found.").


## L2-WORK-003: Move a card between columns
**Traces to:** L1-WORK-001
**Source:** implementation

`POST /api/board/cards/{cardId}/move` with `{ column }` moves a card, subject to the Done lock.

**Acceptance Criteria:**
1. Given a card, when moved to `Backlog`, `InLoop`, or `Review`, then the move succeeds, the card's column updates, and a `CardMoved` event is broadcast.
2. Given a non-existent card id, when moving, then the response is `404`.
3. Given a move to `Done`, when the card's loop is complete, then the card's `currentR` is cleared to null.


## L2-WORK-004: Done column is locked server-side
**Traces to:** L1-WORK-001
**Source:** implementation

The server rejects moving a card into Done unless all five movements are logged.

**Acceptance Criteria:**
1. Given a card with fewer than five movements logged, when moved to `Done`, then the response is `409` (title "The 5R loop is incomplete.") and the card stays in its column.
2. Given a card with all five movements logged, when moved to `Done`, then the move succeeds (`2xx`) and the card's column becomes `Done`.


## L2-WORK-005: Done column is locked client-side
**Traces to:** L1-WORK-001
**Source:** implementation + mocks

The board prevents dropping an incomplete card into Done through a CDK enter-predicate and an
explicit drop guard, and shows the locked Done column with a gate note.

**Acceptance Criteria:**
1. Given a card with `loggedCount < 5` being dragged, when it is over the Done column, then the CDK enter-predicate refuses the drop (the column will not accept it).
2. Given an incomplete card somehow dropped on Done, when `drop()` runs, then no server call is made and a `role="alert"` note reads "{code} can't reach Done until all five movements are logged."
3. Given the board, when it renders, then the Done column is visibly locked and shows a gate note "Needs all 5 R's to enter" with a padlock.
4. Given the server returns `409` for a Done move, when the client receives it, then it shows the same rejection message and reloads the board.


## L2-WORK-006: Card visual state
**Traces to:** L1-WORK-001
**Source:** implementation + mocks

Each work card shows its code, title, a pip strip of 5R progress, its current R as a
develop-green badge, its assignee's initials, and a blocked treatment when applicable.

**Acceptance Criteria:**
1. Given a card, when it renders, then it shows a five-pip strip (labels Rq, Rc, Rv, Rn, Rj) with `i < loggedCount` pips Done and the pip whose kind equals `currentR` marked Current.
2. Given a card with a `currentR`, when it renders, then a `badge--develop` shows that R's name.
3. Given a card with `isBlocked = true`, when it renders, then it carries the blocked (red-border) styling.
4. Given a card, when its title is activated, then the app navigates to `/loop/{cardId}`.


## L2-WORK-007: Retrieve a card's 5R loop
**Traces to:** L1-WORK-002
**Source:** implementation

`GET /api/loop/cards/{cardId}` returns the card's loop with all five movements and a
server-computed `canMarkDone`.

**Acceptance Criteria:**
1. Given a card, when `GET /api/loop/cards/{cardId}` is called, then the response contains `{ cardId, code, title, column, currentR, loggedCount, canMarkDone, movements[] }`.
2. Given each movement, when returned, then it includes `{ id, kind, order, state, ask, received, synthesis, artifactUrl, whatChanged, thanksgiving, loggedAt }`.
3. Given a non-existent card id, when called, then the response is `404`.


## L2-WORK-008: Log the current movement in order
**Traces to:** L1-WORK-002
**Source:** implementation

`POST /api/loop/cards/{cardId}/movements` with `{ kind, ...content }` logs the current R,
records its content, advances the current R, and recomputes movement states.

**Acceptance Criteria:**
1. Given a card whose next R is `Request`, when a movement is logged with `kind = Request`, then that movement's state becomes `Done`, `Receive` becomes `Current`, `loggedCount` becomes 1, and a `MovementLogged` event is broadcast.
2. Given content fields in the request, when a movement is logged, then non-null fields are saved and null fields do not overwrite existing content.
3. Given all five logged in order, when the loop is re-read, then `loggedCount = 5`, `currentR = null`, and `canMarkDone = true`.


## L2-WORK-009: Movements must be logged strictly in order
**Traces to:** L1-WORK-002
**Source:** implementation

Only the current (next unlogged) R may be logged; logging any other R is rejected.

**Acceptance Criteria:**
1. Given a card whose next R is `Request`, when a movement is logged with `kind = Render`, then the response is `400` (title "Movement logged out of order.") and no movement is logged.
2. Given movements logged in the exact order Request â†’ Receive â†’ Review â†’ Render â†’ Rejoice, then each log succeeds.


## L2-WORK-010: Logging pulls a card into the active loop
**Traces to:** L1-WORK-002
**Source:** implementation

Logging the first movement of a Backlog card moves it to InLoop.

**Acceptance Criteria:**
1. Given a card in `Backlog`, when its first movement is logged, then the card's column becomes `InLoop`.
2. Given a card already in `InLoop`, `Review`, or `Done`, when a movement is logged, then its column is unchanged.


## L2-WORK-011: Movement content is R-specific
**Traces to:** L1-WORK-002
**Source:** implementation + mocks

Each R captures its own reflective field(s): Requestâ†’ask, Receiveâ†’received, Reviewâ†’synthesis,
Renderâ†’artifactUrl + whatChanged, Rejoiceâ†’thanksgiving.

**Acceptance Criteria:**
1. Given the current movement is `Render`, when the loop UI renders, then it shows an "Artifact" input and a "What changed from the vision?" field, and logging saves both.
2. Given a logged (Done) movement, when the loop UI renders, then its recorded content is shown as read-only reflective text.
3. Given a future (Locked) movement, when the loop UI renders, then its inputs are disabled/not editable.


## L2-WORK-012: Mark a card Done requires a complete loop
**Traces to:** L1-WORK-002
**Source:** implementation

`POST /api/loop/cards/{cardId}/done` succeeds only when all five movements are logged.

**Acceptance Criteria:**
1. Given a card with fewer than five movements logged, when marking done, then the response is `409` (title "The 5R loop is incomplete.").
2. Given a card with all five logged, when marking done, then the response is `2xx`, the card's column becomes `Done`, `currentR` is null, and a `CardMoved` event is broadcast.


## L2-WORK-013: Mark-done enablement is server-authoritative
**Traces to:** L1-WORK-002
**Source:** implementation + mocks

The loop UI enables "Mark done" strictly from the server's `canMarkDone`; the dial advances from
server truth as movements are logged.

**Acceptance Criteria:**
1. Given a loop at 3/5, when it renders, then the dial reads "3/5" and "Mark done" is disabled.
2. Given the current movement is logged, when the server responds, then the dial advances (e.g. 3/5 â†’ 4/5) from the returned `loggedCount` and "Mark done" stays disabled until 5/5.
3. Given the loop reaches 5/5, when it renders, then "Mark done" is enabled, and activating it navigates back to `/board/{projectId}`.
