# PLAT — Platform Quality: L2 Requirements

Detailed requirements for the [Platform Quality capability](./L1.md). Each requirement traces to one L1 in this capability. Acceptance criteria are in Given/When/Then form; acceptance test files must carry a `Traces to: L2-PLAT-###` comment header.

---

## L2-PLAT-001: String-typed enum contract
**Traces to:** L1-PLAT-001
**Source:** implementation

All enum-valued fields serialize as their names, giving the client a string-typed contract.

**Acceptance Criteria:**
1. Given any response containing an enum (phase kind/state, gate state, column, R kind, movement state), when serialized, then the value is the enum name string (e.g. `"Develop"`, `"Blocked"`, `"Request"`), not a number.
2. Given a request carrying an enum (e.g. move column, log movement kind), when sent as a name string, then the server accepts it.


## L2-PLAT-002: Successful commands return the updated resource
**Traces to:** L1-PLAT-001
**Source:** implementation

Command endpoints return `200` with the resulting resource DTO.

**Acceptance Criteria:**
1. Given a successful create/move/assign, when it completes, then the response is `200` with the updated `CardDto`.
2. Given a successful requirement toggle, when it completes, then the response is `200` with the recomputed `GateDto`; a successful movement log returns the updated `CardLoopDto`.


## L2-PLAT-003: Errors returned as problem+json with correct status
**Traces to:** L1-PLAT-001
**Source:** implementation

Exceptions map to status codes via central middleware and return `application/problem+json`.

**Acceptance Criteria:**
1. Given the failure classes, when they occur, then the status codes are: validation → `400`, out-of-order movement → `400`, invalid credentials → `401`, unauthorized → `401`, email already registered → `409`, incomplete loop → `409`, gate blocked → `409`, card/project/requirement not found → `404`, unexpected → `500`.
2. Given any handled error, when returned, then the body is a `problem+json` document with a `title` (and, where applicable, `detail`).
3. Given an unexpected exception, when it occurs, then the response is `500` with a generic title and the error is logged (no internal details leaked to the client).


## L2-PLAT-004: Health endpoint and deterministic startup
**Traces to:** L1-PLAT-001
**Source:** implementation

The API exposes a health endpoint and, on startup, applies migrations and (in Development) seeds
the Lantern demo data idempotently.

**Acceptance Criteria:**
1. Given the API is running, when `/health` is requested, then it responds successfully.
2. Given a fresh database in Development, when the API starts, then migrations apply and the "New Hope Collective" / Lantern demo data is seeded so the running app matches the mocks.
3. Given the seed runs again on an already-seeded database, when startup occurs, then no duplicate data is created (idempotent).


## L2-PLAT-005: Fast interactive operations
**Traces to:** L1-PLAT-002
**Source:** implementation

Interactive reads and enforcement commands complete quickly under normal load.

**Acceptance Criteria:**
1. Given a signed-in user on typical demo-scale data, when loading a board, journey, or loop, then the API responds in under 300ms at the median (excluding cold start).
2. Given an enforcement command (toggle, log, move), when dispatched, then the enforcement computation itself is in-memory and adds no database round-trips beyond loading the affected aggregate and saving it.


## L2-PLAT-006: Real-time updates arrive promptly
**Traces to:** L1-PLAT-002
**Source:** implementation

Broadcast updates reach other members quickly enough to feel live.

**Acceptance Criteria:**
1. Given two members viewing the same project, when one logs a movement or moves a card, then the other's view reflects it within about one second over the live channel.
2. Given the live channel is degraded, when a member refetches, then they still observe the change (eventual consistency via REST).
